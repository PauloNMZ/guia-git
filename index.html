Guia completo e didático para aprender Git do zero, usando analogias simples

📚Conceitos Básicos: Git vs GitHub

Entender a diferença entre Git e GitHub é fundamental para começar sua jornada.

🔧 Git

Sistema de Controle de Versão
Ferramenta que roda no seu computador. Mantém histórico de todas as mudanças que você faz no seu código. Funciona offline.

☁️ GitHub

Plataforma na Nuvem
Serviço online que hospeda seus repositórios Git. Permite compartilhar código, colaborar e trabalhar em equipe. Funciona apenas online.

📦 Repositório

Pasta do Projeto
Pasta que contém seu projeto e todo o histórico de versões. Pode ser local (sua máquina) ou remoto (GitHub).

Por que aprender Git?

  • ✅ Histórico completo: Veja todas as mudanças já feitas
  • ✅ Colaboração: Trabalhe em equipe sem conflitos
  • ✅ Segurança: Recupere código antigo se necessário
  • ✅ Profissionalismo: Essencial para qualquer desenvolvedor
  • ✅ Ramificação: Trabalhe em múltiplas features simultaneamente

📮Analogia dos Correios

A melhor forma de entender Git é pensar nele como um sistema de Correios. Vamos comparar o fluxo de envio de uma carta com o fluxo do Git!

🔄 O Fluxo Completo: Letra → Envelope → Correios → Destino

Assim como você escreve uma carta, coloca no envelope, leva aos Correios e eles entregam na casa da pessoa, o Git funciona com: adicionar arquivos, fazer commit, enviar para repositório remoto.

📝 Escrever a Carta

Equivalente: Modificar Arquivos
Você modifica seus arquivos de código. Ainda não foi "enviado".

Seus arquivos .js, .py, .html modificados

✉️ Colocar no Envelope

Equivalente: git add
Você prepara os arquivos para envio usando git add

git add arquivo.js
git add .

📮 Levar aos Correios & Lacrar

Equivalente: git commit
Você registra as mudanças com uma mensagem descritiva.

git commit -m "Adicionar novo recurso"

🚀 Enviar para o Destino

Equivalente: git push
Você envia o commit para o repositório remoto (GitHub).

git push origin main
💡 Dica importante: Você pode fazer vários git add e git commit ANTES de fazer git push. É como separar várias cartas antes de levar aos Correios!

🔄Fluxo Completo de Trabalho

Veja passo a passo como um projeto típico é versionado com Git.

1

Criar/Clonar um Repositório

Comece um novo projeto ou clone um existente do GitHub.

git init

ou

git clone https://github.com/usuario/projeto.git
2

Modificar Seus Arquivos

Fazer mudanças no seu código como sempre faz.

Neste momento, o Git já está observando as mudanças!
3

Verificar Status

Veja o que você modificou.

git status
4

Adicionar Arquivos (Staging)

Prepare os arquivos para o commit.

git add .

ou adicione arquivos específicos:

git add src/index.js
5

Fazer o Commit

Registre suas mudanças com uma mensagem descritiva.

git commit -m "Descrição clara do que foi feito"
✅ Bom: "Adicionar validação de email"
❌ Ruim: "alteracoes"
6

Enviar para Remoto (Push)

Envie seu commit para o repositório no GitHub.

git push origin main
⚠️ Importante: Sempre faça git pull antes de começar a trabalhar para atualizar sua máquina local!

⌨️Comandos Essenciais do Dia a Dia

Os comandos que você vai usar 99% do tempo no seu trabalho.

Comando O que faz Exemplo
git init Inicia um novo repositório git init meu-projeto
git clone Copia um repositório remoto git clone <url>
git status Mostra arquivos modificados git status
git add Adiciona arquivos para commit git add . ou git add arquivo.js
git commit Registra as mudanças git commit -m "mensagem"
git push Envia commits para remoto git push origin main
git pull Atualiza repositório local git pull origin main
git branch Cria/lista branches git branch feature-nova
git checkout Muda de branch git checkout feature-nova
git merge Mescla branches git merge feature-nova
git log Vê histórico de commits git log --oneline
git diff Vê diferenças entre versões git diff

Comandos Úteis Avançados

  • git stash - Guarda mudanças temporariamente
  • git reset - Desfaz commits
  • git revert - Reverte um commit mantendo histórico
  • git tag - Cria versões/releases
  • git remote - Gerencia repositórios remotos

⚔️Resolução de Conflitos

Conflitos acontecem quando duas pessoas modificam a mesma linha. Veja como resolver!

🚨 O que é um Conflito?
Ocorre quando você tenta fazer merge de duas branches que modificaram a mesma linha de código.

📋 Cenário Típico de Conflito

Você: Cria uma função chamada calcular()

Seu colega: Modifica a mesma função

Resultado: Conflito ao fazer merge!

1

Git Avisa o Conflito

Você verá uma mensagem como:

CONFLICT (content): Merge conflict in arquivo.js
2

Veja o Arquivo Conflitado

O arquivo mostrará algo assim:

<<<<<<< HEAD
// Seu código aqui
=======
// Código do colega
>>>>>>> branch-colega
3

Edite e Escolha

Abra o arquivo em seu editor e escolha qual código manter. Remova as linhas de marcação do Git.

Você pode manter seu código, o do colega, ou combinar os dois!
4

Commit da Resolução

Adicione e faça commit normalmente:

git add arquivo.js
git commit -m "Resolver conflito em arquivo.js"
✅ Dicas para Evitar Conflitos:
• Faça git pull frequentemente
• Comunique com seu time sobre mudanças
• Use branches para features separadas
• Revise Pull Requests cuidadosamente

🎯Boas Práticas de Git

Siga estas práticas para se tornar um Git ninja!

📝 Mensagens de Commit

Bom: "Adicionar autenticação de 2FA"
Ruim: "fix", "alterações", "atualizacao"
Regra: Use imperativo, seja específico!

🌿 Branches Organizadas

Convenção:
feature/novo-recurso
bugfix/corrigir-erro
hotfix/urgente
Nunca: Trabalhe direto na main!

✅ Commits Pequenos

Faça commits frequentes com mudanças lógicas. Não coloque tudo em um commit gigante!

🔍 Revise Antes de Push

Use git diff e git log para revisar antes de enviar.

🚫 .gitignore

Configure o arquivo .gitignore para não versionar:
node_modules/, .env, arquivos temporários

👥 Pull Requests

Sempre faça Pull Requests, nunca faça merge diretamente. Deixe seu time revisar!

📋 Checklist Antes de Push

💭 Pense assim:

Escreva commits como se estivesse explicando para seu eu do futuro. Seja claro, seja específico, seja profissional!
